> 课程：图机器学习 | 版本：v2.2 | 日期：2026年4月

## 一、系统需求描述

### 1.1 项目背景

传统 RAG（Retrieval-Augmented Generation）系统仅依赖向量语义检索，存在以下局限性：

- **多跳推理能力弱**：无法沿知识关联链进行多步推理（如"周星驰还主演了哪些电影？"需要先找到周星驰，再沿"演员"关系找到所有电影）
- **容易产生幻觉**：当知识库中不存在答案时，大模型倾向于编造信息
- **知识溯源困难**：向量检索返回的文本片段缺乏结构化的来源标注
- **可解释性低**：用户无法理解系统是如何得出答案的

GraphRAG 通过引入**知识图谱**作为结构化知识层，与向量检索形成互补，有效解决了上述问题。

### 1.2 功能需求

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| F1 | 文档上传与处理 | 高 | 支持 PDF/TXT/DOCX/XLSX 多格式文档 |
| F2 | 文本智能分块 | 高 | 使用 LangChain TextSplitter 进行语义切分 |
| F3 | 向量数据库构建 | 高 | ChromaDB + BGE 中文嵌入模型 |
| F4 | LLM 实体关系提取 | 高 | 基于大模型自动提取实体和关系 |
| F5 | 知识图谱构建 | 高 | NetworkX 基础图谱 + Neo4j 增强图谱 |
| F6 | 多跳推理 | 高 | 支持2跳及以上的推理查询 |
| F7 | 混合检索 | 高 | 向量检索 + 图谱检索 + 融合排序 |
| F8 | 知识溯源可视化 | 中 | 推理链路径展示 + 子图可视化 |
| F9 | Prompt 模板管理 | 中 | 系统/NER/RE 三类 Prompt 可视化 |
| F10 | 交互式 Web 界面 | 高 | Streamlit 多标签页界面 |

### 1.3 非功能需求

| 编号 | 需求 | 说明 |
|------|------|------|
| NF1 | 双模式 LLM | 支持 API 模式（DeepSeek/OpenAI）和本地模式（Ollama） |
| NF2 | 自动降级 | Neo4j 不可用时自动回退到内存图谱模式 |
| NF3 | 批量处理 | 支持大批量文档的分批处理，避免内存溢出 |
| NF4 | 实体对齐 | 自动合并同义实体，减少图谱碎片化 |
| NF5 | 社团划分 | Louvain 算法自动检测知识社群 |

## 二、系统设计

### 2.1 系统架构

系统采用**三层架构**设计，高内聚低耦合：

```
┌─────────────────────────────────────────────────────────┐
│                    UI 层 (app.py)                        │
│  Streamlit 多标签页：知识库构建 | 智能问答 | Prompt |    │
│  知识图谱可视化 | 测试案例                                │
└───────────────────────┬─────────────────────────────────┘
                        │
┌───────────────────────┴─────────────────────────────────┐
│                  业务逻辑层 (core/)                       │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ document │  │llm_service│  │vector_store│             │
│  │ 文档处理  │  │ 大模型服务 │  │ 向量数据库 │             │
│  └────┬─────┘  └────┬─────┘  └────┬──────┘              │
│       │              │              │                     │
│  ┌────┴──────────────┴──────────────┴──────┐             │
│  │              neo4j_kg                    │             │
│  │    知识图谱（LLM提取+社团划分+综合查询）  │             │
│  └──────────────────┬──────────────────────┘             │
│                     │                                    │
│  ┌──────────────────┴──────────────────────┐             │
│  │              retriever                   │             │
│  │  检索引擎（向量+图谱+多跳推理+融合排序）  │             │
│  └─────────────────────────────────────────┘             │
└──────────────────────────────────────────────────────────┘
                        │
┌───────────────────────┴─────────────────────────────────┐
│                    数据层                                 │
│  ChromaDB (向量) | NetworkX/Neo4j (图谱) | 文件系统      │
└──────────────────────────────────────────────────────────┘
```

### 2.2 核心模块设计

#### 2.2.1 文档处理模块 (`core/document.py`)

- **功能**：读取多格式文档（PDF/TXT/DOCX/XLSX），文本分块
- **输入**：Streamlit 上传文件对象
- **输出**：文本块列表 `[{'id', 'text', 'source', 'chunk_index'}]`

#### 2.2.2 向量数据库模块 (`core/vector_store.py`)

- **功能**：文本向量化、向量存储、相似度检索
- **技术栈**：ChromaDB + Sentence-Transformers (BAAI/bge-small-zh-v1.5)
- **关键设计**：
  - Excel 数据每行生成一个独立向量，保持事件完整性
  - 非结构化文本使用 LangChain RecursiveCharacterTextSplitter 智能切分
  - 支持余弦相似度检索

#### 2.2.3 大模型服务模块 (`core/llm_service.py`)

- **功能**：LLM 连接管理、Prompt 管理、答案生成
- **双模式设计**：
  - **本地模式**：Ollama + qwen2.5:7b-instruct（免费，无需网络）
  - **API 模式**：DeepSeek/OpenAI API（需网络 + API Key）
  - 自动检测：优先本地，失败回退 API

#### 2.2.4 知识图谱模块 (`core/neo4j_kg.py`)

- **功能**：LLM 实体关系提取、图谱构建、社团划分、综合查询
- **关键设计**：
  - **双 Prompt 模式**：通用模式（20种实体类型+40种关系类型）+ 投诉模式（9种实体+12种关系）
  - **实体归一化**：别名映射表减少同义词导致的图谱碎片
  - **簇间关系构建**：同一投诉人/部门/地点的多条工单自动建立关联
  - **社团划分**：Louvain 算法自动检测知识社群
  - **三层实体链接**：同义词扩展 → 向量相似度 → 图谱关系验证
  - **Neo4j 降级**：连接失败自动回退到内存模式

#### 2.2.5 检索引擎模块 (`core/retriever.py`)

- **功能**：向量检索、图谱检索、混合检索、多跳推理
- **三种检索模式**：
  - **知识图谱检索**：仅使用图谱结构化查询
  - **向量检索**：仅使用语义相似度检索
  - **综合查询**：图谱 + 向量融合（两路并行，取长补短）
- **多跳推理引擎**：自动识别推理模式（HOP_THEN_FIND / FILTER_THEN_MATCH / RELATION_NAVIGATE）

### 2.3 数据流

```
用户上传文档
    │
    ▼
文档处理 (document.py) ──→ 文本块
    │                         │
    ▼                         ▼
向量数据库 (vector_store.py)  LLM实体关系提取 (neo4j_kg.py)
    │                         │
    ▼                         ▼
ChromaDB 向量索引           知识图谱 (NetworkX/Neo4j)
    │                         │
    └─────────┬───────────────┘
              │
              ▼
        检索引擎 (retriever.py)
        ┌─────┴──────┐
        │ 综合查询    │
        │ 向量+图谱   │
        │ +多跳推理   │
        └─────┬──────┘
              │
              ▼
        LLM 答案生成 (llm_service.py)
              │
              ▼
          最终答案 + 溯源
```

### 3.3 关键实现细节

#### 3.3.1 LLM 实体关系提取

系统使用大模型进行联合 NER+RE 提取，而非传统的先 NER 后 RE 的两步法。优势：
- 一次 LLM 调用同时提取实体和关系，减少 API 开销
- 实体和关系可以互相参照，提高一致性
- 支持两种 Prompt 模式切换（通用/投诉）

#### 3.3.2 多跳推理引擎

```python
# 推理模式自动识别
MULTI_HOP_PATTERNS = [
    {"keywords": ["还", "其他", "还有"], "pattern": "HOP_THEN_FIND"},
    {"keywords": ["谁", "哪个", "哪些"], "pattern": "FILTER_THEN_MATCH"},
    {"keywords": ["导演", "演员", "主演"], "pattern": "RELATION_NAVIGATE"},
]
```

推理流程：
1. 识别问题中的核心实体
2. 沿知识图谱关系进行多跳遍历
3. 收集推理路径并构建答案上下文
4. 将推理链注入 LLM 生成最终答案

#### 3.3.3 实体对齐

系统实现了自动化实体对齐策略：
- **字符串包含匹配**：一方完全包含另一方时合并
- **编辑距离匹配**：编辑距离 ≤ 2 且长度 > 3 时合并
- **时间戳保留**：带 `@时间` 的人物实体保留带时间的版本

## 五、核心 Prompt 展示

### 5.1 系统 Prompt（问答生成）

```
你是基于知识图谱的智能问答助手。

【核心原则】
1. 严格基于提供的上下文回答，不要编造信息。信息不足时说"图谱中没有相关信息"
2. 多跳推理：当问题包含"还/其他/以及/另外/和谁"等关键词时，必须执行多步推理
3. 两步推理示例：
   - 问题"周星驰还主演了哪些电影？"
   - 推理：周星驰 --[演员]--> 电影1， 周星驰 --[演员]--> 电影2
   - 回答：周星驰还主演了电影1、电影2等
4. 标注信息来源（知识图谱路径 / 语义相似文档）
5. 回答准确、清晰、有条理

【多跳推理规则】
- 步骤1：找到问题中的核心实体（如人物/电影名）
- 步骤2：沿关系找到关联实体
- 步骤3：再沿关系找到最终答案
- 如果图谱无法支持该推理步骤，明确告知用户
```

## 六、测试报告

### 6.1 测试案例一：简单查询

**问题**：《大话西游》的导演是谁？

| 方法 | 回答 | 准确性 | 说明 |
|------|------|--------|------|
| **纯向量 RAG** | 可能是刘镇伟 | 中 | 返回语义相似的文本片段，但可能包含不确定表述 |
| **GraphRAG** | 刘镇伟 | 高 | 直接从图谱三元组 `《大话西游》--[导演]--> 刘镇伟` 获取，结果确定 |

**GraphRAG 推理过程**：
```
1. 实体识别：提取"大话西游"
2. 图谱查询：MATCH (m:Entity {name: "大话西游"})-[r:导演]->(d:Entity) RETURN d
3. 结果：刘镇伟
```

**分析**：简单查询场景下，两种方法都能找到答案，但 GraphRAG 的结果更加确定和精确，不会出现"可能是"等模糊表述。

---

### 6.2 测试案例二：多跳推理

**问题**：《大话西游》导演的其他代表作有哪些？

| 方法 | 回答 | 准确性 | 说明 |
|------|------|--------|------|
| **纯向量 RAG** | 无法准确回答 | 低 | 向量检索只能找到与"大话西游"语义相似的片段，无法沿"导演"关系跳转到刘镇伟的其他作品 |
| **GraphRAG** | 《都市情缘》、《天长地久》等 | 高 | 两跳推理：大话西游→刘镇伟→其他作品 |

**GraphRAG 推理过程**：
```
第1跳: 《大话西游》 --[导演]--> 刘镇伟
第2跳: 刘镇伟 --[导演]--> 《都市情缘》
第2跳: 刘镇伟 --[导演]--> 《天长地久》
```

**分析**：这是 GraphRAG 的核心优势场景。纯向量 RAG 无法完成多跳推理，因为它缺乏结构化的关系链路。GraphRAG 通过知识图谱的关系遍历，可以轻松实现2跳甚至更多跳的推理。

---

### 6.3 测试案例三：边界案例（无法回答的问题）

**问题**：这部电影的票房是多少？

| 方法 | 回答 | 准确性 | 说明 |
|------|------|--------|------|
| **纯向量 RAG** | 可能编造票房数据（如"该片票房约2亿港元"） | 低 | 大模型倾向于编造看似合理但实际不存在的数据，产生幻觉 |
| **GraphRAG** | 根据提供的知识库，无法回答这个问题 | 高 | 图谱中不存在"票房"相关实体和关系，系统诚实回答"无法回答" |

**GraphRAG 推理过程**：
```
1. 实体识别：提取"这部电影"（指代消解为"大话西游"）、"票房"
2. 图谱查询：MATCH (m:Entity {name: "大话西游"})-[r:*1..2]-(x) WHERE x.name CONTAINS "票房"
3. 结果：空（图谱中不存在票房相关节点）
4. 系统回答：根据提供的知识库，无法回答这个问题
```

**分析**：这是 GraphRAG 避免幻觉的关键能力。当知识库中不存在相关信息时，GraphRAG 能够诚实地回答"无法回答"，而不是编造看似合理的答案。这得益于知识图谱的结构化特性——如果图谱中没有对应的实体和关系，系统可以明确判断信息缺失。

---

### 6.4 综合对比

| 能力维度 | 纯向量 RAG | GraphRAG | 优势说明 |
|---------|-----------|----------|---------|
| 简单查询 | 中 | 优 | GraphRAG 从结构化三元组获取，结果更确定 |
| 多跳推理 | 差 | 优 | GraphRAG 可沿关系链进行多步推理 |
| 避免幻觉 | 差 | 优 | GraphRAG 可明确判断信息是否存在于图谱中 |
| 知识溯源 | 困难 | 清晰 | GraphRAG 可展示完整推理路径 |
| 可解释性 | 低 | 高 | GraphRAG 的推理过程可追溯、可验证 |
| 模糊查询 | 优 | 中 | 向量检索在语义模糊匹配方面仍有优势 |

**结论**：GraphRAG 在结构化查询、多跳推理和避免幻觉方面显著优于纯向量 RAG，而纯向量 RAG 在模糊语义匹配方面仍有其优势。最佳方案是**综合查询模式**，将两种方法融合使用。

## 七、总结与展望

### 7.1 项目成果

1. 完整实现了 GraphRAG 系统的全流程：文档上传 → 知识抽取 → 图谱构建 → 混合检索 → 问答生成
2. 支持双模式 LLM（本地 Ollama + API），降低了使用门槛
3. 实现了多跳推理引擎，支持2跳及以上的推理查询
4. 设计了通用/投诉双 Prompt 模式，适应不同领域
5. 实现了实体对齐、社团划分等高级图谱功能

### 7.2 未来改进方向
